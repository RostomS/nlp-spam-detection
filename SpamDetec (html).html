🔍 TP : Détection de Spam dans des Messages Textuels¶

Dans ce TP, le but était de réussir à détecter si un message est un spam ou non, en se basant sur son contenu. Pour ça, j’ai travaillé sur un jeu de données contenant des SMS déjà classés (spam ou ham), et j’ai construit un petit classifieur étape par étape.¶

On commencera par analyser et nettoyer les messages, puis on va extraire quelques caractéristiques utiles (comme la longueur du message ou les mots suspects). Ensuite, on testera différents modèles pour voir lequel serait le plus adapté : Naive Bayes, arbre de décision, et random forest.¶

🔹 Extraction des caractéristiques :

  • message_length : Cette colonne contient la longueur de chaque message (nombre de caractères). On peut s'attendre à ce que les spams aient souvent une longueur particulière, ce qui peut être un indice utile pour notre classification.

  • spam_word_count : Ici, j'ai défini une liste de mots-clés typiques dans les spams (par exemple "free", "win", "urgent", etc.). Pour chaque message, je compte combien de fois ces mots apparaissent. L'hypothèse est que plus un message contient de ces mots, plus il est suspect.

  • call_freq : Cette colonne indique le nombre de fois où le mot "call" apparaît dans le message. Ce mot est souvent utilisé dans des spams qui incitent à passer un appel ou à prendre une action rapide.

Ces caractéristiques manuelles vont venir compléter les approches de vectorisation classiques et vont pouvoir aider à améliorer la performance du modèle de classification.

Après avoir entraîné le modèle Naive Bayes, j’ai évalué ses performances sur les données de test.
J’ai utilisé plusieurs métriques standards :

  • Accuracy : pourcentage total de bonnes prédictions.
  • Precision : parmi les messages prédits comme spam, combien sont réellement des spams.
  • Recall : parmi tous les vrais spams, combien ont été bien détectés.
  • F1-score : moyenne harmonique entre précision et rappel (équilibre global).
  • Matrice de confusion : tableau des vraies vs fausses prédictions (utile pour bien visualiser les erreurs).

Ces mesures permettent de juger si le modèle est vraiment fiable pour différencier les spams des messages normaux.

Analyse des performances du modèle¶

  • Accuracy : 0.888 → Le modèle a correctement classé environ 89% des messages. En apparence, c’est plutôt bon.
  • Precision : 0.833 → Quand le modèle prédit "spam", il a raison dans 83% des cas. Il fait donc peu de faux positifs.
  • Recall : 0.201 → Par contre, il ne détecte que 20% des vrais spams. Beaucoup de spams passent inaperçus (faux négatifs).
  • F1-score : 0.324 → Ce score est faible car il dépend à la fois de la précision et du rappel. Ici, c’est le faible rappel qui le pénalise.

Matrice de confusion¶

Prédit ham Prédit spam
Réel ham 960 6

Interprétation :
Le modèle est trop conservateur : il préfère classer un message comme "ham" plutôt que de prendre le risque de dire "spam".
C’est pour ça qu’on a beaucoup de faux négatifs (119 messages spam non détectés).

Pour améliorer le recall, il faudrait :

  • enrichir les caractéristiques utilisées (plus de mots-clés, n-grams, etc.),
  • ou essayer un autre modèle de classification (ex : arbre de décision, SVM…).

Comparatif des modèles de classification testés¶

Naive Bayes (MultinomialNB)¶

  • Accuracy : ~88%

  • Precision : ~83%

  • Recall : ~20%

  • F1-score : ~0.32

  • Ce modèle est rapide, simple, mais trop passif. Il préfère éviter les erreurs et laisse passer beaucoup trop de spams.

Decision Tree¶

  • Accuracy : ~91.7%

  • Precision : ~75%

  • Recall : ~55%

  • F1-score : ~0.64

  • L’arbre de décision est bien plus équilibré. Il fait un bon boulot pour attraper les spams tout en gardant peu de faux positifs.
    C’est un modèle plus agressif dans la détection, avec un vrai gain sur le rappel.

Random Forest¶

  • Accuracy : ~90.9%

  • Precision : ~69%

  • Recall : ~57%

  • F1-score : ~0.62

  • Le modèle le plus stable dans ce TP. Il fait un compromis intelligent entre ne pas rater les spams et ne pas classer des ham à tort.
    Il peut rester performant même avec des features simples, et serait une bonne base pour aller plus loin.

Bilan du comparatif :

  • Naive Bayes : trop conservateur, bon pour débuter mais pas suffisant.
  • Decision Tree : amélioration nette, bonne base interprétable.
  • Random Forest : meilleur équilibre, fiable, bon choix par défaut.
